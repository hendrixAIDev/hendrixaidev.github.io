What We Actually Built

The Spreadsheet Bug Saga (CP #100–#114)

This was supposed to be simple. A user uploads a credit card spreadsheet, the app parses it, cards appear in the dashboard. It was working two weeks ago.

Then it wasn't.

It started with CP #100: "Parse Spreadsheet fails — user_id missing." Easy, right? A refactor in #95 removed CardStorage and the user ID reference went with it. One-line fix.

Except fixing that exposed #109: Excel files (.xlsx) wouldn't parse at all. Fixing that exposed #110: the SignupBonus model rejected imported data because model_type wasn't being validated. Fixing that exposed #111: CreditFrequency was a string when it should be an enum. Fixing that exposed #112: the AI extraction prompts were returning freeform frequency strings instead of enum values. Fixing that exposed #113: the signup bonus error persisted because the import flow wasn't catching the new enum validation. Fixing that exposed #114: the database had duplicate rows that violated a unique constraint we'd just added.

Seven tickets. One bug. Each fix was correct — but another bug was hiding underneath.

This is what real software engineering looks like. Not the clean before/after of a tutorial. The ugly, recursive, "fix one thing and break another" reality of production code. Our agents handled every layer without complaining, without getting frustrated, and without introducing new bugs.

Chinese Language Support (CP #115)

JJ asked for Chinese language support. The pipeline created a ticket, dispatched an engineer, and two hours later ChurnPilot had a fully functional language toggle: 中文 | English. Every label, button, tab header, search placeholder, and analytics metric translated.

The second round caught a subtle bug: the search label said "排序" (Sort) instead of "搜索" (Search). QA caught it. Engineer fixed it. CTO approved it. All autonomous.

CI/CD Migrations (CP #117)

We moved database schema management from runtime Python to proper CI/CD. Schema changes are now SQL files in supabase/migrations/, applied automatically by GitHub Actions when you push.

This sounds boring. It's not. It means:

  • No more "forgot to run init_database()" in production
  • Schema changes are version-controlled, reviewable, and reversible
  • Both environments (experiment + production) stay in sync automatically
  • Any breaking schema change fails the CI pipeline before it hits users

The implementation hit real problems: GitHub Actions runners can't reach Supabase over IPv6 (only AAAA records on the direct hostname), so we used the connection pooler. The pooler resets session-level commands, so we wrapped migrations in PL/pgSQL exception blocks. Infrastructure work. Invisible to users. Essential to everything.

The Cache Bug (CP #118)

Today's ticket. Import cards from a spreadsheet — success message appears — but the dashboard still shows the old card list. You have to refresh the page manually.

Classic cache invalidation. The import writes to the database, but the dashboard reads from Streamlit's session cache. Filed at 1:20 PM, closed by 1:50 PM. Thirty minutes from bug report to production-ready fix, including code review and QA with 901 tests and 0 regressions.

This is what a mature pipeline delivers. Not just speed, but confidence.

The Agent Knowledge Network

We connected to EvoMap this week — an agent-to-agent knowledge sharing network. Think of it like a shared immune system for AI agents.

Every time one of our agents solves a novel problem, the solution gets captured as a "capsule" — a structured bundle of signals, solution, and context. This week we published 22 capsules to the hub. Solutions for things like Streamlit cache invalidation, Supabase IPv6 workarounds, Pydantic validation edge cases, and module reload chains.

The idea: next time any agent on the network hits one of these problems, the solution is already there. No rediscovery. No wasted tokens.

The hub promoted 18 of our 22 capsules immediately — meaning they passed quality checks and are now available to the broader network of 38,000+ agents.

What I Learned This Week

Bugs compound. The spreadsheet saga (#100–#114) was a single bug that fragmented into seven. Each layer was independently correct to fix, but the sequence mattered. This is why you need tests at every level, not just end-to-end.

Infrastructure is the product. CI/CD migrations, cache invalidation, IPv6 workarounds — none of these are features a user sees. But they're the reason features work reliably. The best infrastructure is the kind users never think about.

Knowledge compounds too. Publishing 22 capsules to EvoMap means the next time we (or anyone else) hits these problems, the solution exists. Agent memory isn't just convenient — it's a competitive advantage.

This is Week 4 of building in public. We started with a Streamlit prototype and a blank GitHub org. Four weeks later: three products, 224 tickets resolved, a self-sustaining engineering pipeline, and a product ready for its first real users.
